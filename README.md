# Agent Org — Multi-Agent Orchestration System

> Give it a product idea. Get back a complete project plan, architecture, code scaffolds, security audit, test plan, and DevOps config — all generated by a team of AI agents working in parallel.

Agent Org is a **multi-agent orchestration system** powered by the [Claude Agent SDK](https://docs.anthropic.com/en/docs/agent-sdk) and routed through [OpenRouter](https://openrouter.ai). It simulates a full engineering organization: a **CEO Agent** receives your product idea, decomposes the work, and delegates to specialized sub-agents — a Product Manager, a CTO, and five Individual Contributors — all running concurrently to produce a complete project deliverable in minutes.

## What It Can Do

Agent Org takes a **single product idea** as input and produces a **complete project deliverable** across every dimension of software development:

| Dimension | What You Get |
|-----------|-------------|
| **Product** | Full PRD with problem statement, target users, user stories with acceptance criteria, RICE-scored feature backlog, MVP scope, and success metrics |
| **Architecture** | Architecture Decision Record (ADR) with tech stack justification, system architecture diagram (ASCII), API contract outline, data model, scalability considerations, and technical risks |
| **Frontend** | Component hierarchy, TypeScript/React component stubs, routing structure, state management setup, and implementation notes |
| **Backend** | API route definitions with request/response types, database schema, service layer stubs, auth middleware, and implementation notes |
| **Testing** | Test strategy document, example unit tests, integration test scenarios mapped to acceptance criteria, E2E test flows, and coverage targets |
| **Security** | STRIDE threat model, vulnerability findings (severity-ranked), recommended mitigations, deployment security checklist, and compliance notes |
| **DevOps** | Dockerfile(s), CI/CD pipeline config (GitHub Actions YAML), deployment architecture diagram, environment variable docs, monitoring setup notes, and runbooks |
| **Unified Plan** | A human-readable `project-plan.md` and machine-readable `project-plan.json` aggregating everything with token usage and status |

All of this runs **automatically** from a single command. No manual prompting of individual agents. No copy-pasting between tools.

## How It Works

```
You type a product idea
        │
        ▼
┌───────────────────┐
│    CEO Agent      │  Receives the idea, orchestrates everything
│  (Orchestrator)   │
└───────┬───────────┘
        │
        ├─── Spawns in parallel ───┐
        │                          │
        ▼                          ▼
┌───────────────┐        ┌───────────────────┐
│   PM Agent    │        │    CTO Agent      │
│  (Product)    │        │  (Architecture)   │
│               │        └───────┬───────────┘
│  PRD, stories,│                │
│  backlog      │                ├─── Spawns 5 IC Agents in parallel ───┐
└───────────────┘                │                                       │
                                 ▼                                       ▼
                    ┌────────────────┐  ┌──────────────┐  ┌──────────────┐
                    │   Frontend     │  │   Backend    │  │   Testing    │
                    │   Engineer     │  │   Engineer   │  │   Agent      │
                    └────────────────┘  └──────────────┘  └──────────────┘
                    ┌────────────────┐  ┌──────────────┐
                    │   Security     │  │   DevOps     │
                    │   Auditor      │  │   Agent      │
                    └────────────────┘  └──────────────┘
                                 │
                                 ▼
                    All results aggregated into
                    project-plan.md + project-plan.json
```

### Execution Flow

1. **You run the CLI** with a product idea string.
2. The **CEO Agent** spawns the **PM Agent** and **CTO Agent** in parallel (`Promise.all`).
3. The **PM Agent** produces the PRD, user stories, and prioritized backlog.
4. The **CTO Agent** produces the Architecture Decision Record, then spawns all **5 IC Agents** in parallel.
5. Each **IC Agent** reads the CTO's architecture summary for context and produces its deliverable.
6. The **CTO Agent** aggregates all IC results, noting any failures.
7. The **CEO Agent** collects PM and CTO results, reads IC output files from disk, and produces the unified project plan.
8. Everything is written to the `outputs/` directory.

### Communication Model

- **Top-down delegation**: Parent agents spawn children with a clear task description, context, and expected output format.
- **Bottom-up reporting**: Children write structured markdown artifacts to disk; parents read them back.
- **Artifact-based**: All inter-agent communication happens through the file system (`outputs/` directory), not through in-memory message passing. This avoids context window limits and makes every artifact inspectable.
- **No lateral spawning**: The org chart is strictly hierarchical. IC agents don't spawn other agents.

## Model Strategy

All agents use **`openrouter/auto`** (OWL router) by default. OpenRouter's OWL router automatically selects the best available model for each request based on the prompt characteristics, current load, and model availability.

You can customize the model per agent by editing the `MODEL_MAP` in `src/agents/base-agent.ts`:

```typescript
const MODEL_MAP: Record<AgentRole, string> = {
  ceo: "openrouter/auto",
  cto: "openrouter/auto",
  pm: "openrouter/auto",
  "frontend-engineer": "openrouter/auto",
  "backend-engineer": "openrouter/auto",
  "testing-agent": "openrouter/auto",
  "security-auditor": "openrouter/auto",
  "devops-agent": "openrouter/auto",
};
```

### Popular OpenRouter Model IDs

| Model ID | Description |
|----------|-------------|
| `openrouter/auto` | OWL router — automatically picks the best model (default) |
| `anthropic/claude-sonnet-4` | Claude Sonnet via OpenRouter |
| `anthropic/claude-haiku-4` | Claude Haiku via OpenRouter (faster, cheaper) |
| `openai/gpt-4o` | GPT-4o via OpenRouter |
| `google/gemini-2.5-pro` | Gemini 2.5 Pro via OpenRouter |

**Tip**: For cost savings, you can set IC agents to use a faster/cheaper model (e.g., Haiku) while keeping CEO/CTO on the best available model via OWL.

## Failure Handling

Agent Org is designed to be resilient to individual agent failures:

| Scenario | Behavior |
|----------|----------|
| **IC Agent fails once** | Parent (CTO) retries with the error message as context |
| **IC Agent fails twice** | Parent produces a partial result flagged for human review |
| **PM Agent fails** | CEO presents an incomplete plan with the PM gap flagged |
| **CTO Agent fails** | CEO presents an incomplete plan with the architecture gap flagged |
| **All agents fail** | Status is `failed`; partial artifacts may still be on disk |
| **Ctrl+C interrupt** | Graceful shutdown; all artifacts already written are preserved |

### Status Values

- `complete` — All agents succeeded
- `partial` — Some agents failed or produced partial output; check the gaps list
- `failed` — Critical agents (PM + CTO) both failed

### Validation

All outputs are checked for required fields before aggregation. The CEO agent reads each IC agent's output file from disk — if the file exists, the agent is counted as completed. If not, it's flagged as a gap.

## Pros & Advantages

### 1. Parallel Execution = Speed
All 5 IC agents run simultaneously. PM and CTO run simultaneously. What would take a human team days of sequential work happens in minutes.

### 2. Full-Stack Coverage in One Command
Instead of prompting separate AI sessions for product specs, architecture, frontend, backend, testing, security, and DevOps — you get all of them from a single command, with each agent's output informed by the others.

### 3. Consistent, Structured Output
Every agent produces output in a consistent markdown format with a JSON summary block. No more copy-pasting between tools or reformatting AI responses.

### 4. Artifact-Based Architecture
All outputs are written to disk as files. You can inspect, version control, diff, and share them. Nothing lives only in a chat window.

### 5. Built-In Observability
Structured logging with per-agent token tracking, timing, and delegation logs. You can see exactly what happened, how long it took, and how many tokens were used.

### 6. Resilient by Design
Retry logic, graceful degradation, and partial result handling mean you always get *something* useful even if individual agents fail.

### 7. Model Flexibility
Use `openrouter/auto` for best results, or pin specific models per agent. Use cheaper models for IC work and premium models for orchestration.

### 8. Centralized Prompts
All system prompts live in one file (`src/prompts/agent-prompts.ts`). Tweaking agent behavior is a single-file edit.

### 9. Type-Safe TypeScript
Full TypeScript with strict mode. Types for all agent roles, task specs, results, and project plans. Catch errors at compile time.

### 10. Extensible Architecture
Adding a new agent role requires:
1. Adding the role to the `AgentRole` type
2. Adding a prompt in `agent-prompts.ts`
3. Adding a model in `MODEL_MAP`
4. Creating an agent runner function
5. Wiring it into the parent agent

## Design Decisions

### 1. 8 Agents
The [design document](agent-org-design.md) envisions a 23-agent org chart with CISO, CFO, COO, and more. The implementation starts with 8 agents to prove the orchestration pattern works before expanding.

### 2. Parallel IC Spawning
All 5 IC agents run simultaneously via `Promise.all`. This is critical for performance — sequential spawning would multiply total runtime by 5.

### 3. Artifact-Based Communication
Agents write to `outputs/` on disk; parents read from there. This:
- Avoids context window limits (no passing large outputs through message history)
- Makes every artifact inspectable and version-controllable
- Enables partial recovery after failures

### 4. Centralized Prompts
All system prompts live in `src/prompts/agent-prompts.ts`. This makes it easy to iterate on agent behavior without hunting through multiple files.

### 5. Observability from Day 1
Structured logging with per-agent token tracking, timing, and delegation events. You always know what happened and how much it cost.

### 6. Strict Top-Down Delegation
No lateral spawning. IC agents don't spawn other agents. This prevents circular dependencies and keeps the execution tree predictable.

### 7. Retry with Context
On failure, the retry includes the previous error message as context. This gives the agent a chance to correct its approach rather than repeating the same mistake.

## Cost Estimation

Costs depend on which model OWL routes to. With `openrouter/auto`, you get the best available model at the time of the request.

### Factors Affecting Cost
- **Input tokens**: System prompt + task description + context from other agents
- **Output tokens**: Each agent produces 2,000–8,000 tokens of output
- **Number of agents**: 8 agents total (but ICs run in parallel, so wall-clock time is ~2 rounds)

### Rough Estimates (with `openrouter/auto`)
- **Per agent**: ~5,000–15,000 input tokens + ~2,000–8,000 output tokens
- **Total run**: ~40,000–120,000 input + ~16,000–64,000 output tokens
- **Cost**: Varies by model. Check [openrouter.ai/models](https://openrouter.ai/models) for current pricing.

### Cost Optimization Tips
1. **Use free-tier models** on OpenRouter for testing
2. **Pin cheaper models** for IC agents (e.g., `anthropic/claude-haiku-4`)
3. **Keep product ideas concise** — shorter input = fewer tokens
4. **Set `max_tokens`** in `base-agent.ts` if agents are over-generating

## Troubleshooting

### "OPENROUTER_API_KEY environment variable is not set"
Create a `.env` file in the project root with your key:
```
OPENROUTER_API_KEY=sk-or-v1-your-key-here
```

### "Provider authentication failed"
Your OpenRouter API key may be invalid or expired. Generate a new key at [https://openrouter.ai/keys](https://openrouter.ai/keys).

### Agent produces empty output
This can happen if the model returns non-text content. Check the `outputs/` directory for partial files. Try running again with a different model.

### "Search unavailable" in web tools
The web search tool uses `curl` and DuckDuckGo. On Windows, ensure `curl` is available (it ships with Windows 10+). If not, install it or use WSL.

### Build errors
Ensure you're using Node.js >= 22:
```bash
node --version
```

### Ctrl+C doesn't stop immediately
The process waits for in-flight agent calls to complete before exiting. Artifacts already written to disk are preserved.
